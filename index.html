LogCatcher

A tiny, reliable Android logcat viewer for macOS and Windows.

The short version. I'm an Android developer. For years I used LogRabbit to follow adb logcat while debugging. It's not a bad app — but it crashed often, and updates stopped coming. After one too many mid-debug crashes cost me a repro I had to chase for an hour, I gave up waiting.

LogCatcher is the smallest viewer that does the things I actually need: connect to a device, filter the noise, keep my colors and fonts, remember my filter groups, and stay out of the way. No accounts, no cloud, no telemetry, no surprises.

Why it might be worth your time

  • Native Cocoa on macOS, native Win32 on Windows — fast launch, low memory.
  • Real-time streaming with smart auto-scroll (sticks to the bottom unless you scroll up).
  • Rich filter syntax: app:, tag:, level:, pid:, tid:, message:, AND / OR, parentheses.
  • Saved filter groups with rename → Save-As semantics.
  • Per-level text and background colors, fully customizable.
  • Incremental search with ⌘F / Ctrl+F, “N of M” status, prev/next.
  • Log-buffer mode switch (“From Now” vs. “All Buffer”) with tunable tail lines.
  • Works with any adb on your PATH — no vendoring, no version drift.

Manual

1. Prerequisites

LogCatcher does not bundle the Android platform tools. You need adb from the Android SDK Platform Tools, either on your PATH or in one of the standard install locations (LogCatcher checks ~/Library/Android/sdk/platform-tools, /opt/homebrew/bin, /usr/local/bin, %LOCALAPPDATA%\Android\Sdk\platform-tools, etc.).

To verify:

adb devices

If that lists your device, LogCatcher will too.

2. First launch on macOS

Because the app is ad-hoc signed rather than notarized, a fresh machine may show “app is damaged and can't be opened.” It isn't damaged — macOS just hasn't seen a signature it recognizes. Two options:

  • DMG: open Terminal once and run xattr -cr /Applications/LogCatcher.app.
  • PKG: nothing to do — the installer clears the quarantine flag for you.

3. Connecting a device

Plug in your device (or start an emulator), pick it from the toolbar's device dropdown, and click Connect. If the dropdown is empty, click Refresh. If exactly one device is connected, LogCatcher picks it automatically; if several, it remembers your last choice.

4. Reading logs

Rows are color-coded by level. Auto-scroll sticks to the bottom of the buffer while new lines arrive. Scroll up to pause auto-scroll; scroll back to the bottom to resume it — no toggle needed. When the buffer fills its cap and the oldest entries are pruned, the visible rows stay anchored where you left them — the view doesn't “jump forward.” Double-click a row to open a detail window with Copy Message and Copy All.

The toolbar carries a Tail / All toggle right next to Auto-scroll. Click to flip between:

  • Tail — fastest start; only the recent tail of the device buffer is pulled on connect (line count configured in Preferences).
  • All — full device log buffer. Slower initial fill but complete history.

The button label itself shows the current mode (or T / A in icon-mode). It mirrors the Preferences Log buffer mode setting both directions — change either, the other follows.

5. Filter syntax

Type into the filter field at the top of the window. Bare words match anywhere in the row. Prefixes scope the match:

PrefixExampleMeaning
app:app:com.exampleProcess package name
tag:tag:ActivityManagerLog tag
level:level:ELevel (V, D, I, W, E, F)
pid:pid:1234Process ID
tid:tid:1234Thread ID
message:message:ANRMessage body only

Combine terms with AND / OR. Parentheses group. Examples:

level:E AND app:com.example
tag:ActivityManager AND (level:W OR level:E)
message:ANR AND NOT app:com.android.systemui

Uppercase the operators (AND, OR, NOT); lowercase is treated as a literal word.

6. Search within the buffer

Press ⌘F (macOS) or Ctrl+F (Windows) to open the find field. Type to highlight matches; the status reads “N of M”. Use Next / Previous (or Enter / Shift+Enter) to jump. Esc closes the field.

7. Saved filter groups

Open Settings → Filter Settings… to manage saved filters. Create a group, name it, and paste in your filter expression. To edit an existing group, change its name or query and click Save — the old entry is renamed in place. To fork a group into a new one, change the name and click Save As.

8. Global Filter (always-applied exclude list)

Some entries are noise you never want to see — kernel pid:0 spam, your own analytics tag, etc. Open Settings → Global Filter… for a single, hidden filter group that's applied after any saved filter you have selected: anything matching the Global Filter is dropped from the view, regardless of which saved filter is active.

  • The default seed when you first open it is pid is 0 — kernel-level rows that almost no one wants.
  • The Global Filter group is named ~exclusion and intentionally hidden from the main filter list and the Filter Manager so you can't accidentally pick it like a regular filter. The only way to edit it is the Global Filter button.
  • Empty the condition list to disable the Global Filter — an empty group is a no-op.
  • Edits use the same condition editor as any saved filter; the only difference is the name field is locked.

9. Preferences

Open from the app menu (LogCatcher → Settings… on macOS, Options → Preferences on Windows). The dialog scrolls when needed; the bottom action buttons stay visible.

  • Filters — Filter Settings… manages saved groups; Global Filter… opens the always-applied exclude list (see section 8).
  • Log Level Colors — per-level text and background color. Click a swatch to open the system color picker.
  • Log Font — monospace font + size for the table.
  • Message Column Width — initial width (200–5000 px). The column is resizable at runtime too.
  • General
    • Use icons for toolbar buttons — compact toolbar mode (single-letter labels).
    • Log buffer mode: Tail (recent tail only, faster start) or All (full device buffer). Mirrors the toolbar Tail/All toggle.
    • Tail recent lines — how many tail lines to pull on connect (1–100 000).
    • ADB connection (macOS) — pick how LogCatcher reaches the device. See section 10.

10. ADB connection modes (macOS)

The mac build can reach an Android device three ways. Pick one in Settings → ADB connection; the change takes effect on the next Connect.

ModeWhat it doesWhen to use
USB (direct)   default Talks directly to the device over USB using IOKit + the ADB packet protocol. No adb binary or server needed. Day-to-day use. Bypasses adb-server entirely; needed for the App-Store-friendly path. First Connect on a new computer triggers the “Allow USB debugging?” prompt on the phone — tap Allow.
TCP (adb server) Connects to a running adb server on 127.0.0.1:5037 and speaks the textual server protocol. The server is auto-started if missing. When you want to share the device with Android Studio or other adb-based tools without re-pairing.
Subprocess (adb binary) Forks the adb binary for every operation (legacy behavior). Diagnostic / fallback if either of the above misbehaves on your setup.

USB direct caveats:

  • If adb-server or Android Studio is holding the device, LogCatcher will seize the interface — those tools may then briefly lose the device. Switch back to TCP/Subprocess to share.
  • Authentication uses your existing ~/.android/adbkey. If the device hasn't seen this key before, the phone will prompt for permission — you have ~60 seconds to tap Allow.
  • USB direct is mac-only for now; Windows always uses TCP/Subprocess.

13. Troubleshooting

  • No devices listed. Run adb devices from a terminal. If that fails, adb isn't on your PATH or the device isn't authorized (unlock the phone and accept the USB-debugging prompt).
  • USB direct (mac): “USB interface is held exclusively.” Android Studio or another tool has the device. Quit it, or switch to TCP/Subprocess.
  • USB direct: “Device rejected our key” / prompt never appears. Run adb kill-server then adb devices once with the phone unlocked, accept the USB-debugging dialog, then retry USB direct.
  • Logs show “?” in the app column. The ADB process map is fetched asynchronously after connect. It fills in within a second or two.
  • macOS: “app is damaged.” See First launch on macOS above — run xattr -cr or install via the PKG.
  • Windows: SmartScreen warning. The build is unsigned. Click More info → Run anyway. Only do this if you trust the source.
